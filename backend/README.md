# Система автоматического назначения дежурных — Backend

Бэкенд REST API для управления дежурствами в учебных группах вуза. Система автоматически назначает дежурных с учётом рейтинга, отслеживает отсутствия и переназначает дежурства в режиме реального времени.

## Описание

Приложение предназначено для старост учебных групп. Каждый пользователь (староста) создаёт свои группы, добавляет в них студентов и настраивает расписания дежурств. Система автоматически каждый день в 06:00 генерирует события дежурств, выбирая студентов по принципу **минимального рейтинга (duty_score)** — дежурит тот, кто дежурил меньше всего и чаще пропускал без уважительной причины.

**Ключевые принципы:**
- Дежурным назначается студент с **наименьшим `duty_score`** среди активных участников группы.
- Студент, который отсутствует без уважительной причины в свой день, получает **штраф** (уменьшение `duty_score`), а на его место автоматически ставится замена.
- Одобренное отсутствие не влечёт штрафа — студент просто пропускает свой день без последствий.
- Система гарантирует, что один студент не может быть назначен на два дежурства в один день в рамках одной группы.

## Модули и сущности

### Auth — Аутентификация

Реализует **беспарольную аутентификацию** через одноразовые коды подтверждения (OTP), отправляемые на email или телефон.

**Сервисы:**
- `AuthService` — генерация и верификация JWT-токенов, управление refresh токенами и blacklist-ом отозванных access токенов. Все данные хранятся **в памяти** (`Map`), автоматическая очистка просроченных записей каждые 10 минут.
- `VerificationService` — генерация 6-значных OTP-кодов, хранение в памяти (Map), автоматическая очистка просроченных кодов каждую минуту. Коды действительны **5 минут**.

**Токены:**

| Токен | Время жизни | Описание |
|---|---|---|
| `accessToken` | 15 минут | JWT, передаётся в заголовке `Authorization: Bearer` |
| `refreshToken` | 30 дней | Случайная строка (64 байта), хранится хэш SHA-256 в памяти |

**Механизм refresh (Token Rotation):**
1. При запросе `POST /users/refresh` старый refresh токен уничтожается.
2. Выдаётся новая пара: свежий `accessToken` + новый `refreshToken`.
3. Повторное использование старого refresh токена возвращает `401`.

**Механизм logout:**
1. Refresh токен удаляется из хранилища.
2. Access токен добавляется в blacklist до момента его истечения.
3. Повторное использование access токена возвращает `401`.

> **Примечание:** Отправка кодов (email/SMS) заготовлена как TODO. В режиме разработки код выводится в консоль сервера.

> ⚠️ Поскольку refresh токены хранятся в памяти, при перезапуске сервера все сессии сбрасываются — пользователям потребуется войти заново.

### DutyEvents — События дежурств

Конкретный факт назначения студента на дежурство в определённую дату. Является основной операционной единицей системы.

**Сущность `DutyEvent`:**

| Поле | Тип | Описание |
|---|---|---|
| `id` | number | Первичный ключ |
| `student_id` | number | FK → Students |
| `schedule_id` | number | FK → DutySchedules |
| `duty_date` | date | Дата дежурства |
| `status` | enum | Статус события (см. ниже) |
| `score_earned` | number (nullable) | Изменение очков за это событие |
| `notes` | string (nullable) | Заметки/причина |
| `assigned_at` | timestamp | Дата создания назначения |

**Статусы событий:**

| Статус | Кто устанавливает | Описание |
|---|---|---|
| `pending` | Система | Назначено, ожидает выполнения |
| `completed` | Пользователь | Выполнено, начисляется `+default_score` |
| `cancelled` | Пользователь | Отменено, очки не меняются |
| `reassigned` | Система | Переназначено из-за отсутствия студента |

**Допустимые переходы статусов:**
```
pending → completed     (+score)
pending → cancelled     (score не меняется)
completed ↔ cancelled   (score корректируется с откатом)
reassigned              (финальный, изменить нельзя)
```

**Сущность `Absence`:**

| Поле | Тип | Описание |

| `date_from` | date | Начало отсутствия |
| `date_to` | date | Конец отсутствия |
| `reason` | string (nullable) | Причина отсутствия |
| `is_approved` | boolean | Одобрено ли (влияет на штраф) |

---

### Scheduler — Планировщик

Cron-сервис `DutySchedulerService` запускает авто-генерацию событий дежурств каждый день в **06:00**.

**Логика работы:**
1. Определяет текущий день недели.
2. Выбирает все активные расписания, у которых текущий день отмечен в `DutyDays` и текущая дата входит в период `start_date … end_date`.
3. Для каждого расписания запускает алгоритм назначения студентов.
4. Если события на сегодня уже существуют — пропускает (идемпотентность).

Также доступна **ручная генерация** через API для конкретного расписания и произвольной даты.

## Бизнес-логика

### Алгоритм назначения дежурных

Для каждого слота дежурства (количество слотов = `students_per_day`):

1. Загружаются все **активные** студенты группы.
2. Студенты, уже имеющие `pending`-событие по другому расписанию группы **на эту же дату**, исключаются (один студент — не более одного дежурства в день в группе).
3. Все оставшиеся студенты сортируются по `duty_score` по возрастанию (первым — тот, кто дежурил меньше всего).
4. Перебираем отсортированную очередь:
   - Если студент **доступен** → создаётся событие `pending`, слот закрыт.
   - Если студент **отсутствует** → создаётся событие `reassigned`:
     - Если отсутствие **не одобрено** → начисляется штраф `−default_score`.
     - Если отсутствие **одобрено** → очки не меняются.
     - Затем ищется **замена** — первый доступный студент с минимальным `duty_score`, которому создаётся `pending`.
5. Вся операция выполняется в рамках одной транзакции БД.

### Статусы событий и очки (duty_score)

```
Выполнено (pending → completed):
  student.duty_score += duty_type.default_score
  event.score_earned  = +default_score

Отменено (pending → cancelled):
  student.duty_score — без изменений
  event.score_earned  = null

Переназначено системой (reassigned):
  Если отсутствие НЕ одобрено:
    student.duty_score -= duty_type.default_score
    event.score_earned  = -default_score
  Если отсутствие одобрено:
    student.duty_score — без изменений
    event.score_earned  = null

Смена статуса completed ↔ cancelled:
  Откатывается ранее начисленный score_earned
  Применяется новое значение
```

### Обработка отсутствий

При создании или обновлении записи об отсутствии автоматически запускается `handleAbsenceUpsert()`:

1. Находит все `pending`-события студента в диапазоне `date_from … date_to`.
2. Для каждого такого события:
   - Переводит событие в статус `reassigned` (со штрафом, если отсутствие не одобрено).
   - Ищет замену: активный студент той же группы, не отсутствующий и не занятый в эту дату, с минимальным `duty_score`.
   - Создаёт для замены новое событие `pending`.

## API и документация

После запуска приложения интерактивная документация Swagger UI доступна по адресу:

```
http://localhost:3000/api
```

Все защищённые маршруты требуют заголовок:
```
Authorization: Bearer <ACCESS_TOKEN>
```

**Типичный сценарий работы с токенами на клиенте:**
```
1. POST /users/login        → сохранить accessToken и refreshToken
2. Любой запрос             → Authorization: Bearer <accessToken>
3. Получен 401              → POST /users/refresh { refreshToken }
                              → сохранить новую пару токенов, повторить запрос
4. POST /users/logout       → Authorization: Bearer <accessToken>
                              + body: { refreshToken }
```

**Группы эндпоинтов:**

| Тег | Базовый путь | Описание |
|---|---|---|
